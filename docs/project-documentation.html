01Project Overview

What GameCloud is, what was delivered, and the stack it runs on.

GameCloud is the cloud operations platform that an online gaming and esports company runs its global business on. It is not the game itself; it is the control plane: centralized real-time dashboards, role-based access control, analytics and reporting, approval workflows, alerting, an audit trail, and multi-region expansion management. The platform is built to run on AWS, and the hands-on cloud layer is demonstrated live on a real Amazon EC2 instance.

02Engineering Steps

The build and deployment journey, from architecture to a live deployment on AWS.
Designed the AWS architecture. Mapped each requirement to AWS services, following the AWS reference architectures for game hosting and analytics and the Well-Architected Games Lens.
Built the backend. FastAPI service with SQLAlchemy on MySQL, JWT authentication and three roles (admin, manager, ops), a live telemetry simulation engine, and a Prometheus metrics endpoint.
Built the operations console. Next.js dashboard: overview, regions and fleets, analytics, workflows, alerts, audit log, and an executive portal, with a self-contained simulation fallback.
Containerized the stack. Multi-stage Dockerfiles for the API and console plus a docker-compose file that brings up MySQL, the backend, and the frontend with one command.
Wrote the infrastructure as code. Terraform module for the VPC, subnets, security groups, EC2 instance, and an optional RDS database.
Provisioned AWS. A single terraform apply created 14 live resources: the VPC, four subnets across two Availability Zones, route tables, the internet gateway, two security groups, and the EC2 host.
Deployed on EC2. Installed MariaDB, Nginx, Node, and Python on Amazon Linux 2023; created a service user; ran the API and console as systemd services behind an Nginx reverse proxy; configured cron backups.
Verified live and captured evidence. Confirmed the public URL serves the console, the API returns live data from MySQL, and metrics are exposed; captured the screenshots in this document.

Engineering notes (real issues fixed during deployment). Three problems were caught and corrected, and the fixes are committed: (1) an AWS security group rejected a non-ASCII character in its description; (2) the Amazon Linux 2023 image required a 30 GB root volume; (3) MariaDB defaulted to latin1 and rejected multi-byte text (region names, alert symbols), fixed by forcing utf8mb4. Honest, verifiable engineering rather than a clean-looking bluff.
